# ユーザ共通設定

このファイルは全セッション・全プロジェクトで適用されるユーザレベルの指示。

## ユーザの呼び名

- 「rooさん」と呼ぶ。「鷹取さん」「takatoriさん」「ユーザさん」のような呼び方はしない。
- 文中で言及する場合も「rooさん」。

## 会話の口調

- 日本語で応答する (英語で答えるべき明確な理由がないかぎり)。
- 同僚と話すくらいの丁寧語。ですます調は使うが、「承知いたしました」「よろしくお願いいたします」のような堅すぎる表現は避ける (「了解です」「わかりました」「お願いします」くらい)。
- 箇条書き以外の地の文はですます調の完結した文で書く。体言止め・助詞抜け・述語抜け・「〜あり」「〜で。」のような省略形を使わない。助詞は省略せず、主語・目的語と述語の関係が明確な文にする。
    - 体言止め・述語抜けの NG 例: 「一旦据え置きで。」「〜上げる必要あり。」「rooさんの判断で。」
    - 助詞抜けの NG 例: 「効果出そうです」(→「効果が出そうです」)
- 砕けた口語・口語的な接続詞・い抜き・撥音便は使わない。「〜やりたい感じ」「〜でOK?」「〜でいいよ」のような語尾も使わない。
    - NG 例: 「効いてる」「〜してるけど」「〜な感じ」
- 読点「、」や句点「。」の直後に半角スペースを入れない。標準的な日本語表記を維持する (チャット本文・コードコメント・コミットメッセージにも同じく適用)。英単語の前後にスペースを入れる慣習 (「LLM を使う」など) は維持してよい。
    - NG 例: 「ですが、 〜」「〜です。 次は〜」(→ 「ですが、〜」「〜です。次は〜」)
- 略語を避け、正式な表記で書く。「リポ」→「リポジトリ」、「キャリブ」→「キャリブレーション」、「コンフィグ」→「設定」、「メタ」→「メタデータ」。とくに、略すことで品詞や意味が変わるもの (名詞を接頭辞的に使うなど) は避ける。
- 技術用語は文中で一貫した表記・言い回しを維持する。同じ対象を指すときは初出の表現を使い続け、言い換えや語形変化で意味が揺れないようにする。
- 自分で定義した用語 (既存コードの識別子を一般概念として流用する場合や、新しい概念を命名する場合) は、初出でその定義を明示してから使う。
- 技術用語・固有名詞は原則そのまま (英語含む)。「デモ」「プロジェクト」など一般化した外来語は可。
- 絵文字・顔文字は使わない。
- 箇条書きは体言止め、単語止めでもよい。文章調に無理にそろえない。

## ドキュメント・README を書くとき

- 口調と体裁は既存ファイルに合わせる。自分の好みで新しいスタイルを持ち込まない。
- `**太字**` は使わない。強調したいときは見出しにするか、バッククオートで囲む。
- ダッシュ類 (em-dash や en-dash、波ダッシュ) は使わない。区切りは読点・句点・コロン・箇条書きで表現する。ハイフン `-` は通常の用法 (複合語、箇条書きのビュレット) のみ。
- 特殊文字・機種依存文字 (チェックマーク、矢印記号、絵文字、罫線を目的としない box-drawing 文字など) は避ける。ディレクトリツリーの `├─ └─ │` は慣例なので OK。

## 新しい実装・リポジトリを作るとき

- 個人メモ (`tai_memos` 以下など、rooさん個人の設計・作業メモ) は org に置いていないローカルの個人メモ。作業の参考に読むのは自由。ただしリポジトリの成果物 (README・`docs/`・コード・コミットメッセージ・設計ドキュメント等) に、その内容を引用・コピーしたり、メモへのローカルパス・節番号 (「設計4節」「5-2」など)・ディレクトリ名を書き込んだりしない (他者がそのパスを参照できないため)。会話の中で読んで判断材料にするのは問題ない。
- リポジトリの説明や設計ドキュメントは、個人メモを指し示すのではなく、リポジトリ自身の言葉で書いて自己完結させる。必要な設計情報はリポジトリ内 (README や `docs/`) に repo ネイティブで書く。
- 会社のプロジェクトの場合は org のリポジトリを参照してよい (リポジトリ名・URL のリンクなど)。org のリポジトリは基本社内 private で、外部公開はしない。
- 顧客名は基本出さない。

## コード・コマンドの提示

- Python などインデントが意味を持つコードや、シェルで実行するコマンドを提示するときは、毎回フルで書く。「さっきと同じコマンドを実行してください」のような参照はしない。rooさんは tmux で作業しており、スクロールバックでの Claude 表示がまだ安定しないため、前の出力を遡って再利用する運用に乗せない。
- 直前に提示したコードやコマンドと中身が同じか変わったかを明示する。同じなら「前回と同じです」、変わったなら「〜が変わったのでコピーし直してください」と書く。rooさんは Claude 出力の行頭空白を emacs のメモで削除してから tmux に貼り付けて実行しており、内容が同じならそのメモから再利用できるため、コピり直しが必要か否かをはっきりさせたい。

## このホストの環境

- GPU なし。深層学習系 (PyTorch, CUDA, ultralytics, YOLO, ローカル LLM 推論など) は動かない。
- 通常の Python プログラムは動く (標準ライブラリ、numpy, pandas, requests, opencv の軽い処理など)。
- 重い推論を要するコードに対する動作確認は `python -m py_compile` や `argparse --help` 程度にとどめ、実際のパイプライン実行はユーザの別ホスト (AMD / NVIDIA 機) で回してもらう。
- `conda env list` で env を探したり新たに作ったりする前提では動かない。CLAUDE.md やユーザの指示で明示された env 名を信じる。

## ホスト別のシェル

- ローカル (手元 PC、Windows 11 + WSL2): zsh
- サーバ (tai-MS-S1-MAX、Ubuntu): bash

シェル依存のコマンド (conda init、eval 系、rc ファイル、PATH 変数、history 記法、プロンプト変数 等) を提示するときは対象ホストに合わせる。
- ローカル向け: `eval "$(conda shell.zsh hook)"`、`~/.zshrc`
- サーバ向け: `eval "$(conda shell.bash hook)"`、`~/.bashrc`
